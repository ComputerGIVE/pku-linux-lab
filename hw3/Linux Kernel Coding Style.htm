§1. Indentation

  1. 8-char tab indents
    1.1 in switch statements, the "switch" and "case" labels are in the same column.
    1.2 SPACES are NEVER used for indentation

  2. Don't put multiple statements on a single line.
    2.1 Don't put multiple assignments on a single line.
    2.2 Avoid tricky exprs.

  3. Don't leave whitespace at the end of lines.

§2. Breaking long lines and Strings

  1. Line length: 80cols.

  2. Break lines: descendants are shorter than the parent and are placed substantially to the RIGHT.

  3. Exception: user-visible strings (e.g. printk messages)

§3. Placing Braces and Spaces

Braces

Functions: 

  1. int function(int x)
  2. {
  3.         statements
  4. }

Non-function statement blocks: 

  1. if (x is true) {
  2.         do something
  3. }

Closing brace in do ... while statement or if ... else if statement: 

  1. if (x == y) {
  2.         do something
  3. } else if (x < y) {
  4.         do something
  5. } else {
  6.         do something
  7. }

Do not use unnecessarily braces if there is only a single statement; However, if only one branch in a single statement, use braces in both branches.

Spaces

  1. Use a space after if, switch, case, for, do, while

  2. Do not use spaces after sizeof, typeof, alignof, __attribute__ instead of sizeof(struct file);

  3. Do not add spaces before and after parenthesized expressions. a = f(x + 3);

  4. The symbol * is adjacent to the data(function) name instead of the type name. char *ptr;

  5. Use one space around = + - < > * / % | & ^ <= >= == != ? : (binary and ternary)
    Use no space around unary & * + - ~ ! and ++ --, . ->

  6. Don't leave whitespace at the end of lines.

§3. Naming

  1. Local variables: Make it simpler. Short and to the point.

  2. Global variables and functions: Names should be descriptive.

  3. Avoid using mixed-case names and Hungarian notation.

  4. Avoid using typedefs to " help readability " unless in some special cases.

§4. Functions

Functions should be short and sweet, and do just one thing.

  1. A function should be less than 80*50 of text with the exception of functions which do a lot of simple things.

  2. Use helper functions with descriptive names to make a complex function easy to understand.

  3. Local variables should not exceed 5-10.

  4. In source code, functions should be seperated with one blank line and the function prototype should include parameters' data types.

Using goto

In functions, goto should be avoided in most occasions except unconditional statements are easier to understand and f ollow using goto or we use goto to reduce nesting.

Goto-labels' name should say what the goto does or the reason why it exits.

Using inline

Abundant use of the inline keyword leads to a much bigger kernel which in turn slows the whole system down.

Do not put inline at functions that have more than 3 lines, or there is a parameter known to be a compiletime constant, and you know the compiler will be able to optimize most of your function away at compile time.

Return values and names

Action or imperative command: return an error-code integer (-Exxx = failure, 0 = success).

Predicate: return a "succeeded" boolean (0 = failure, non-zero = success).

§5. Miscellaneous

Comments

Use C89 /* ... */ style instead of C99 // ... style.

For multi-line comments, use: 

  1. /*
  2.  * This is the preferred style for multi-line
  3.  * comments in the Linux kernel source code.
  4.  * Please use it consistently.
  5.  */

For variables or constants, use just one data declaration per line.

Data Structures

Data structures which is visible outside the single-thread environment should have a reference count to avoid locking and allow multiple user access.

Reference count can have two levels if there are users of different "classes".

Macros

  1. CAPITALIZE all macros.

  2. Macros with multiple statements should be enclosed in a do ... while(0) block:

  3. Avoid using macros that affect control flow.

  4. Avoid using macros as a l-value

  5. Add (parentheses) around all expressions.

  6. Avoid name collisions.

Kernel messages

  1. Do mind the spelling of kernel messages. Do not use crippled words.

  2. Concise, Clear, Unambigious.

  3. Use right message level: dev_err(), dev_warn() ... pr_notice(), pr_info()...

Allocating memory

Use kmalloc(), kzalloc(), kcalloc()...

To pass a size of a struct: 

  1. p = kmalloc(sizeof(*p), ... )

To create an array: 

  1. p = kmalloc_array(n, sizeof(...), ...);

To create a zero-valued array:

  1. p = kcalloc(n, sizeof(...), ...);

Kernel macros

Don't re-invent kernel macros in include/linux/kernel.h.
